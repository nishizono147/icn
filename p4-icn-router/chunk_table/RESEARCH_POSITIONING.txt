================================================================================
  先行研究（port 511 版マルチチャンク ICN ルータ）と本研究の位置づけ
  研究発表・論文執筆用メモ
================================================================================

【このメモの目的】
  先行研究を「間違っていた」とは言わず、同一 BMv2 環境上での
  「プロトタイプ → E2E 検証に基づく完成」として本研究を位置づけるための
  言い回し・構成・ Q&A をまとめたもの。


--------------------------------------------------------------------------------
1. 前提の整理（何が起きていたか）
--------------------------------------------------------------------------------

  先行研究
    - BMv2（simple_switch）上でマルチチャンク ICN 中継ルータを実装
    - キャッシュヒット時に Interest 1 本から Data N 本を返すため
      I2E clone + clone session egress port 511 を用いる構成
    - port 511 は「内部ループ専用 egress」として想定

  判明したこと（本研究の検証過程で）
    - BMv2 simple_switch では port 511 = default_drop_port（破棄ポート）
    - Interest クローンが drop され、中継スイッチからの N チャンク配信が
      End-to-End では成立していなかった
    - 先行段階では Cold パス（Producer 経由の初回取得）中心の評価で、
      Warm パス（中継キャッシュヒット → N Data）の E2E 確認が不十分だった

  本研究
    - 同一設計目標（中継 ICN キャッシュ + マルチチャンク配信）を維持
    - clone + egress recirculate（Interest のみ内部ループ）に再構成
    - flag 制御による段階的キャッシュ配置を統合
    - image4（4 チャンク）で Cold / キャッシュ移動 / Warm を E2E 検証

--------------------------------------------------------------------------------
3. 3 段ストーリー（発表・論文の骨子）
--------------------------------------------------------------------------------

  【第 1 段：先行研究の位置づけ（評価）】

    先行研究では、BMv2 上にチャンク分割 ICN ルータのプロトタイプを構築し、
    register によるチャンクキャッシュ、およびキャッシュヒット時に
    Interest 1 本から Data N 本を生成する必要性を示した。
    内部ループには I2E clone と egress port 511 による方式が採用された。

    → 貢献：問題設定・プロトコル・実装方針の提示

  【第 2 段：残っていたギャップ（批判ではなく「未検証・未完了」）】

    評価は主に Cold パス（Producer 応答）に焦点が当たっており、
    Warm パス（中継キャッシュヒット時の N チャンク返却）の
    End-to-End 確認が不十分であった。
    より厳密な検証（register 状態、pcap、Warm フェーズ、h2 非到達確認）により、
    clone egress 先 port 511 が BMv2 では drop ポートであり、
    中継からの複数チャンク配信が成立していなかったことが判明した。

    → ギャップ：同一 BMv2 上での E2E 保証と実装具体化

  【第 3 段：本研究の貢献（拡張・完成）】

    本研究は、先行研究と同一の設計目標を維持し、
    BMv2 向けに clone + egress recirculate 方式へ再構成した。
    Interest クローンのみ recirculate し Data は Consumer へ送出する分岐を明確化し、
    flag 制御による段階的キャッシュ配置を統合した。
    image4（1024 B / 4 チャンク）で Cold → キャッシュ移動 → Warm を検証した。

    → 貢献：BMv2 上で E2E 動作する完成形 + 実装知見

8. 質疑応答の想定
--------------------------------------------------------------------------------

  Q: 同じ BMv2 なのに、なぜ先行時に気づかなかった？

  A: 先行段階の評価は主に Producer 経由の初回取得（Cold パス）に
     焦点が当たっており、中継スイッチ register の全チャンク確認や
     Warm 時の Producer 非到達など、インルーターキャッシュ配信に
     特化した E2E テストが十分でなかった。
     本研究では評価を Cold / キャッシュ移動 / Warm に分離し、
     その過程で port 511 問題を特定した。

  ---

  Q: それは先行研究の失敗では？

  A: 設計方針（マルチチャンク中継キャッシュ + clone 内部ループ）自体は
     有効である。未完了だったのは BMv2 上での具体実装の E2E 検証である。
     本研究は同じ環境・同じ目標に対する完成形を示した、と答える。

  ---

  Q: port 511 はなぜダメだったの？

  A: BMv2 simple_switch では port 511 が drop ポートであるため、
     内部ループ用 egress として機能しない。
     clone による「1 Interest → N Data」という方針は維持し、
     egress recirculate により BMv2 上で実現した。

9. 論文・スライド向け文案（コピペ用）
--------------------------------------------------------------------------------

  【背景・動機（1 段落）】

    先行研究では BMv2 上にチャンク分割 ICN 中継ルータのプロトタイプが
    構築され、キャッシュヒット時の複数 Data 生成には clone による
    内部ループが用いられた。しかし中継スイッチからの End-to-End 検証を
    深化させる過程で、clone session の egress 先 port 511 が
    BMv2 simple_switch において drop ポートであることが判明し、
    中継キャッシュヒット時の N チャンク配信が成立していなかった。

  【本研究の位置づけ（1 段落）】

    本研究は、先行研究と同一の設計目標（Interest 1 本から register 上の
    N チャンクを Consumer へ返す）を維持し、BMv2 向けに
    clone と egress recirculate の組み合わせへ再構成した。
    これにより、同一検証環境上で中継 ICN キャッシュからの
    複数チャンク配信を初めて End-to-End で確認した。

  【貢献（箇条書き）】

    1. BMv2 simple_switch における clone / recirculate の適切な組み合わせ
    2. Interest と Data の分岐による Consumer への誤送出防止
    3. flag 制御型段階的キャッシュ配置とマルチチャンク配信の統合
    4. image4 による Cold / 移動 / Warm の自動 E2E 検証

  【タイトル案】

    - BMv2 向けマルチチャンク ICN 中継キャッシュの E2E 検証と実装完成
    - clone 内部ループの BMv2 向け再構成：中継 ICN キャッシュの複数チャンク配信
