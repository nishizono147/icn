================================================================================
  先行研究（port 511 版マルチチャンク ICN ルータ）と本研究の位置づけ
  研究発表・論文執筆用メモ
================================================================================

【このメモの目的】
  先行研究（BMv2 エミュレーション上で「中継ルータから複数チャンク配信できた」
  と公表済み）と本研究の関係を、矛盾を隠さず、かつ「全面否定」にもならない
  叙述で整理するための言い回し・構成・Q&A をまとめたもの。

【重要な前提】
  先行研究は pit_table ではなく port 511 版マルチチャンク ICN ルータ。
  同一 BMv2 上でエミュレーションし、「中継から複数チャンク配信に成功した」
  と報告している。
  本研究の厳密再検証では、その Warm パス（中継キャッシュヒット → N チャンク
  E2E）は成立していなかった（port 511 = drop 等）。
  → 「単なる拡張」だけでは公表内容との矛盾を説明できない。
  → 「再評価 → 問題特定 → 修正実装 → E2E 再確認」の線で語る。


--------------------------------------------------------------------------------
1. 前提の整理（何が起きていたか）
--------------------------------------------------------------------------------

  先行研究（公表済み）
    - BMv2（simple_switch）上でマルチチャンク ICN 中継ルータを実装
    - キャッシュヒット時に Interest 1 本から Data N 本を返すため
      I2E clone + clone session egress port 511 を用いる構成
    - port 511 は「内部ループ専用 egress」として想定
    - **エミュレーション結果として、中継ルータからの複数チャンク配信に
      成功したと報告している**

  本研究の再検証で判明したこと
    - BMv2 simple_switch では port 511 = default_drop_port（破棄ポート）
    - Interest クローンが drop され、中継スイッチからの N チャンク配信は
      **Consumer が全チャンクを受信・再構成できる条件** では E2E 成立しない
    - 先行報告時の評価は Cold パス中心、または clone ログ・部分的送出等を
      もって成功と判断していた可能性（当時の成功判定基準の洗い出しが必要）
    - 同一 BMv2 環境であり「プラットフォームを知らなかった」は使えない

  本研究
    - 先行報告の **再現性検証** を実施
    - 問題特定（port 511）のうえ **修正実装**（clone + egress recirculate）
    - flag 制御による段階的キャッシュ配置を統合
    - image4（4 チャンク）で Cold / キャッシュ移動 / Warm を E2E 自動検証
    - **厳密な成功条件**（register 全チャンク、h2 非到達、MD5 一致等）を満たす
      初めての BMv2 上の中継 N チャンク配信


--------------------------------------------------------------------------------
2. 公表済み主張との矛盾 — どう扱うか
--------------------------------------------------------------------------------

  【矛盾の中身】
    公表：「BMv2 エミュレーションで中継ルータから複数チャンク配信できた」
    再検証：「同条件 E2E（Warm、Consumer 再構成）では成立していなかった」

  【取れない方針】
    × 「本研究は先行の拡張です」だけで矛盾を無視する
    × 黙って「改良版」だけ出し、先行の成功主張と矛盾する記述を並置する
    × 「先行は嘘だった」「全面否定」とだけ言う

  【推奨する方針：再評価 → 修正 → 再検証】
    ○ 先行報告の **再現性検証** を本研究の動機の一つとする
    ○ **評価指標の厳密化**（当時 vs 今回）を明示する
    ○ **修正実装** により同一目標を E2E で達成した、と述べる
    ○ 可能なら先行公表への **追記・訂正・学位論文での補足** を検討する

  【「拡張」と「再評価」の使い分け】
    拡張だけ     … 公表済み「できた」と矛盾しやすい
    再評価+修正  … 公表内容と正面から整合できる

3. 言い方の原則（避ける / 使う）
--------------------------------------------------------------------------------

  【避ける】
    × 「先行研究は間違っていた」「嘘だった」
    × 「port 511 の選択は誤りだった」（単純化しすぎ）
    × 「BMv2 を知らなかったから失敗した」（同じ BMv2 を使っていたため不誠実）
    × 「中継キャッシュ自体ができていなかった」（設計・実装・報告は存在した）
    × 「単なる拡張で解決する」だけの説明（公表主張との矛盾を隠す）

  【使う】
    ○ 「先行研究では BMv2 上で中継からの複数チャンク配信を報告した」
    ○ 「本研究では **再現性検証と評価条件の厳密化** を行った」
    ○ 「報告されていた成功は、**E2E 成功条件**（全チャンク・Consumer 再構成）
        では満たされていないことが判明した」
    ○ 「設計目標（1 Interest → N Data）は妥当。**E2E を満たす BMv2 実装**
        を本研究で提示した」
    ○ 「port 511 方式では BMv2 上の中継 N チャンク E2E を保証できないことを
        示し、recirculate 方式で **同一目標を E2E 確認** した」
    ○ 「当時の評価指標と本研究の評価指標の差を Discussion で整理した」

5. 3 段ストーリー（発表・論文の骨子）— 「拡張」から「再評価+修正」へ
--------------------------------------------------------------------------------

  【第 1 段：先行研究の位置づけ（公表内容を認める）】

    先行研究では、BMv2 上にチャンク分割 ICN 中継ルータを実装し、
    register キャッシュ、clone による内部ループ（port 511）、
    および **中継ルータからの複数チャンク配信** をエミュレーションで
    **実現したと報告** している。

    → 貢献：問題設定・プロトコル・実装方針・初期評価

  【第 2 段：本研究の契機 — 再現性検証と評価の厳密化】

    本研究ではこの結果を **独立に再現・検証** するため、
    register 状態、pcap、Warm フェーズ、Producer 非到達、
    Consumer 側ファイル再構成（MD5）等を含む **E2E 成功条件** を定義した。
    その結果、clone egress 先 port 511 が BMv2 drop ポートであり、
    **公表されていた中継 N チャンク配信は、当該 E2E 条件では
    成立していない** ことが判明した。
    先行報告時点の成功判定は、当該 E2E 条件に対して **過大評価** だった。

    → ギャップ：評価指標の厳密化により見えた実装要因（511）

  【第 3 段：本研究の貢献 — 修正実装と E2E 再確認】

    本研究は設計目標（Interest 1 本 → register から N Data）を維持し、
    clone + egress recirculate 方式へ **再構成** した。
    flag 制御による段階的キャッシュ配置を統合し、
    image4 で Cold / キャッシュ移動 / Warm を **自動 E2E 検証** した。
    **Consumer 再構成まで含めた中継 N チャンク配信を、
    同一 BMv2 上で E2E 確認した初の報告** として位置づける。

    → 貢献：再評価 + 問題特定 + 修正実装 + 厳密 E2E 検証

11. 質疑応答の想定
--------------------------------------------------------------------------------

  Q: 先行では「中継から配信できた」と言っているのに、矛盾しない？

  A: 本研究では Consumer による **完全再構成**（全チャンク・MD5）等を
     E2E 成功条件として **再現検証** した。
     その条件では port 511 実装は Warm パスが成立せず、
     先行報告の成功判定は **当該 E2E 条件に対しては過大評価** だった。
     本研究は **修正実装** により、同一 BMv2・同一目標で
     E2E 条件を初めて満たした、と答える。
     （当時何をもって成功としたかは表で明示する）

  ---

  Q: 同じ BMv2 なのに、なぜ先行時に気づかなかった？

  A: 先行段階の評価指標が E2E 成功条件まで及んでおらず、
     clone ログや Cold パス等をもって全体が成功していると
     判断していた可能性がある。
     本研究では評価を Cold / 移動 / Warm に分離し、
     register・pcap・MD5 まで含めた **再現性検証** を行った。

  ---

  Q: それは先行研究の失敗では？

  A: 設計方針（clone による 1 Interest → N Data）は有効である。
     問題は **E2E 成功条件を満たす BMv2 具体化** と **その検証** にあった。
     本研究は再評価・修正・再確認の過程である、と答える。
     「全面否定」ではなく「評価の深化と実装の完成」。

  ---

  Q: port 511 はなぜダメだったの？

  A: BMv2 simple_switch では port 511 が drop ポート。
     内部ループ用 egress として機能しない。
     clone 方針は維持し、egress recirculate で E2E 達成した。

12. 論文・スライド向け文案（コピペ用）
--------------------------------------------------------------------------------

  【背景・動機（1 段落）】

    先行研究では BMv2 上にチャンク分割 ICN 中継ルータを構築し、
    中継ルータからの複数チャンクコンテンツ配信をエミュレーションにより
    実現したと報告している。本研究ではこの結果の再現性を検証するため、
    Consumer による全チャンク受信とコンテンツ再構成（MD5 一致）等を含む
    End-to-End 成功条件を定義した。その再現検証の結果、clone session の
    egress 先 port 511 が BMv2 simple_switch において drop ポートである
    ため、当該 E2E 条件では中継キャッシュヒット時の N チャンク配信が
    成立していないことが判明した。

  【本研究の位置づけ（1 段落）】

    本研究は、先行研究と同一の設計目標を維持しつつ、BMv2 向け実装を
    clone と egress recirculate の組み合わせへ再構成した。さらに flag 制御
    による段階的キャッシュ配置を統合し、定義した E2E 成功条件の下で
    image4（4 チャンク）の Cold・キャッシュ移動・Warm を自動検証した。
    これにより、Consumer 再構成まで含めた中継 ICN キャッシュからの
    複数チャンク配信を、同一 BMv2 環境で E2E 確認した。

  【貢献（箇条書き）】

    1. 先行報告に対する再現性検証と E2E 成功条件の明示
    2. port 511 実装が BMv2 上で Warm E2E を満たさないことの特定
    3. clone / recirculate による修正実装と Interest/Data 分岐
    4. flag 制御型段階的キャッシュとマルチチャンク配信の統合
    5. image4 による自動 E2E 検証（run_chunk_experiment.py）

  【タイトル案】

    - 中継マルチチャンク ICN キャッシュの再評価と BMv2 向け E2E 実装
    - BMv2 における clone 内部ループの再構成：中継 ICN 複数チャンク配信の
      再現性検証と修正

  【先生・査読向け説明（1 文）】

    先行公表時点では BMv2 上で中継からの複数チャンク配信が動作していると
    判断・報告していたが、本研究で E2E 条件まで含む再現検証を行った結果、
    その判断は過大評価であった。port 511 問題を特定し recirculate 方式で
    初めて同条件 E2E を満たした。
